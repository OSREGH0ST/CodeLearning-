🧱 Object-Oriented Design

Beyond the Basics

💡 Architectural Context

Deep inheritance trees lead to the "Gorilla Banana" problem (you request a banana, but you get a gorilla holding the banana and the entire jungle). Modern OOP strongly favors Composition over Inheritance. Build systems using small, interchangeable interfaces rather than rigid base classes.

Encapsulation (State Management)

Not just making fields private. It is about protecting system invariants. Objects should never expose setters that allow them to be put into an invalid state. Mutate state only through intention-revealing methods.

Polymorphism via Interfaces

Rely on defining behaviors (what an object can do) rather than taxonomies (what an object is). This enables Dependency Injection and facilitates decoupled unit testing via mocks/stubs.

Composition Example (TypeScript)
TypeScript 
// Bad: Inheritance
class Bird { fly() {} }
class Penguin extends Bird { fly() { throw new Error("Cannot fly"); } } // Liskov Violation

// Good: Composition (Interfaces/Traits)
interface Swimmer { swim(): void; }
interface Flyer { fly(): void; }

class Penguin implements Swimmer {
    swim() { console.log("Swimming fast!"); }
}

The SOLID Principles

PrincipleCore ConceptSenior Context
S: Single Responsibility A class should have one, and only one, reason to change. Separates UI logic, business rules, and persistence. Reduces merge conflicts and blast radius during refactors.
O: Open/Closed Entities should be open for extension, but closed for modification. Use Polymorphism/Strategy patterns. You should be able to add new features by adding new files, not mutating existing if/switch statements.
L: Liskov Substitution Subtypes must be substitutable for their base types. If it looks like a duck, quacks like a duck, but needs batteries, you probably have the wrong abstraction.
I: Interface Segregation Clients should not be forced to depend on interfaces they do not use. Role interfaces. A Printer interface should not force a BasicPrinter class to implement a fax() method.
D: Dependency Inversion High-level modules should not depend on low-level modules. Both should depend on abstractions. The core of Hexagonal Architecture / Clean Architecture. Inject dependencies (DB connections, API clients) via constructors.

Tactical Domain-Driven Design

Entities

Objects defined primarily by their identity, which persists over time (e.g., User, Order). Two entities with identical properties but different IDs are distinct.

Value Objects

Objects defined by their attributes, with no conceptual identity (e.g., Money, Coordinates, Color). They must be immutable.

Aggregates & Root Entities

Clusters of domain objects treated as a single unit. An Aggregate Root ensures the consistency of changes being made within the aggregate boundary. External objects should hold references only to the Root, not to internal members.

Object Lifecycle Management

💡 Anti-Pattern Warning

Avoid the Singleton pattern unless strictly necessary (e.g., hardware spoolers, specific loggers). It introduces global state, tightly couples code, and makes unit testing exceptionally difficult. Use Dependency Injection to manage single-instance lifecycles instead.

Factory Method

Defines an interface for creating an object, but lets subclasses alter the type of objects that will be created. Great for abstracting complex instantiation logic.

Builder

Separates the construction of a complex object from its representation. Highly useful when an object requires multiple configuration steps or parameters (preventing the "telescoping constructor" anti-pattern).

Composition & Wrappers

PatternUse CaseAnalogy
Adapter Allows incompatible interfaces to collaborate. Wraps an existing class with a new interface. A USB-C to USB-A dongle.
Decorator Attaches new behaviors to objects dynamically by placing them inside wrapper objects. Putting on a jacket, then a raincoat. Base identity remains, behavior expands.
Facade Provides a simplified interface to a complex subsystem of classes, libraries, or frameworks. A car's steering wheel and pedals hiding the complex engine internals.

Algorithms & Communication

Strategy

Defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime. Eliminates massive conditional blocks.

Observer

A subscription mechanism to notify multiple objects about events happening to the object they're observing. (Foundational for Reactive UI frameworks).

Command Pattern

Turns a request into a stand-alone object containing all information about the request. Enables parameterization of methods, delayed execution, queuing, and undoable operations.

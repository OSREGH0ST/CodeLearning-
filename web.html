Modern Rendering Architectures

💡 Architectural Context

Default to Server-Side Rendering (SSR) or Static Site Generation (SSG) for content-heavy applications. Single Page Applications (SPAs) are best reserved for highly interactive, dashboard-like interfaces where hydration overhead is justified by continuous client-side state mutations.

Server-Side Rendering (SSR)

HTML is generated on each request. Optimal for SEO and Time to First Byte (TTFB). Frameworks: Next.js, Nuxt, SvelteKit.

Static Site Generation (SSG)

HTML is built at compile time. Near-instant load times via CDN delivery. Suitable for blogs, documentation, and marketing pages.

Client-Side Rendering (CSR)

A bare HTML skeleton is shipped; JavaScript builds the UI in the browser. High initial load time, fast subsequent navigation.

Incremental Static Regeneration (ISR)

Allows updating static content post-build without full site rebuilds. Combines the speed of SSG with the freshness of SSR.

State Management & Core Web Vitals

Server State vs. Client State

Do not mix transient UI state (e.g., dropdown toggles) with asynchronous server cache. Use specialized tools for data fetching (e.g., React Query, SWR) to handle caching, background refetching, and deduping, leaving tools like Redux or Zustand strictly for global UI state.

Hydration & Optimization

Hydration attaches event listeners to server-rendered HTML. To avoid the "Uncanny Valley" (visible but non-interactive UI):

  • Partial Hydration / Islands Architecture: Only hydrate interactive components (e.g., Astro).
  • Code Splitting: Delay loading non-critical JavaScript until idle or on interaction.
  • Tree Shaking: Eliminate dead code during the build process using bundlers like Vite or Webpack.

Distributed Systems Architecture

💡 Architectural Context

Monoliths are not an anti-pattern. Start with a modular monolith. Extract to microservices only when independent deployment, scaling, or varying tech stacks become a business requirement, as microservices introduce severe operational complexity (network partitions, eventual consistency).

Communication Protocols
  • REST: Resource-oriented, relies on standard HTTP methods (GET, POST, PUT, DELETE). Uses standard HTTP caching.
  • GraphQL: Single endpoint. Solves over-fetching/under-fetching. Complex to cache and secure against deep query attacks.
  • gRPC: Uses Protocol Buffers (Protobuf) over HTTP/2. Strongly typed, highly efficient binary serialization. Ideal for service-to-service internal communication.

Data Persistence

Relational (SQL)

PostgreSQL, MySQL. Enforce ACID (Atomicity, Consistency, Isolation, Durability). Use when data integrity, strict schemas, and complex joins are required.

NoSQL (Document/Key-Value)

MongoDB, DynamoDB, Redis. Often favor BASE (Basically Available, Soft state, Eventual consistency). Use for unstructured data, high write throughput, or caching.

Caching Strategies
Redis Cache-Aside Pattern 
async function getUser(userId) {
  // 1. Check cache
  const cached = await redis.get(`user:${userId}`);
  if (cached) return JSON.parse(cached);

  // 2. Cache miss: fetch from DB
  const user = await db.query('SELECT * FROM users WHERE id = $1', [userId]);
  
  // 3. Populate cache with TTL (Time To Live)
  if (user) {
    await redis.set(`user:${userId}`, JSON.stringify(user), 'EX', 3600);
  }
  
  return user;
}

AppSec Fundamentals

VulnerabilityMitigation Strategy
SQL InjectionAlways use parameterized queries or an ORM/Query Builder. Never concatenate user input directly into SQL strings.
XSS (Cross-Site Scripting)Sanitize user inputs. Use proper HTTP headers (Content-Security-Policy). Modern frontend frameworks escape inputs by default.
CSRF (Cross-Site Request Forgery)Implement Anti-CSRF tokens. Use SameSite=Strict attribute on session cookies.
Auth ArchitectureStore JWTs in HttpOnly, Secure cookies—not in LocalStorage. Hash passwords using Bcrypt or Argon2.
